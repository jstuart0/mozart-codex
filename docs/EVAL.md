# EVAL — mozart evaluating mozart

EVAL is the fourth work shape: mozart examines the campaign artifacts its past runs left behind in your projects, measures how well the pipeline actually performed, verifies whether the previous eval's fixes worked, and proposes configuration improvements. Invoke it with the `mozart-eval` skill (or "run a mozart eval" through the `mozart` skill).

The authoritative pipeline definition lives in the **EVAL pipeline** section of `.codex/skills/mozart/SKILL.md`. This document covers the artifacts: where they live, the ledger schema, and the report format.

## Eval home

Eval artifacts span projects, so they live in a user-scope directory — never inside a consuming repo, and never inside the installed port (which is read-only for port users):

1. `$MOZART_EVAL_HOME`, if set
2. `~/.mozart/evals/` (default; created on first use)

Nothing in the eval home is committed anywhere by default. It is your private operational record — repo names, paths, and campaign details stay on your machine.

## The ledger (`<eval-home>/ledger.jsonl`)

Append-only JSONL, machine-written (generated by a loop or script during stage 6 — never hand-edited). One record per (run, repo, slug, lens):

```json
{"run":"2026-07-06","repo":"my-project","repo_path":"/abs/path/to/my-project","slug":"2026-06-20-deliver-some-feature","layout":"finished","state_md5":"<md5 of the state file>","lens":"inventory"}
```

Fields:

| Field | Meaning |
|---|---|
| `run` | ISO date of the eval run this record belongs to |
| `repo` | Short repo name (basename is fine; must be stable across runs) |
| `repo_path` | Absolute path of the canonical checkout examined |
| `slug` | Campaign slug (bare — no `active-`/`finished-` prefix) |
| `layout` | Where the state file was found: `active` \| `finished` \| `active-prefix` \| `finished-prefix` \| `flat` |
| `state_md5` | Hash of the state file at examination time — the delta key |
| `lens` | What was asked of this campaign: `inventory` (enumerated, lifecycle/hygiene-checked) \| `execution-quality` (deep-read: gate value, catch attribution, stall forensics) \| any custom lens name a later run introduces |

Semantics:

- **Delta**: a campaign is in-scope for the next run when its slug is absent from the ledger or its current `state_md5` differs from the newest record.
- **Revisit-by-lens**: an unchanged campaign may be re-examined under a lens no prior record carries. The new record documents that the question has now been asked — this is how deeper understanding of already-seen artifacts stays cheap and deliberate rather than accidental re-reading.
- **Corrections are new records**, not rewrites. The newest record for a (repo, slug, lens) wins.
- **Canonical checkouts only.** Worktree replicas of the same repo are excluded; a state file that diverges between checkouts is a finding for the report, not a ledger entry per copy.

## The report (`<eval-home>/<YYYY-MM-DD>-eval.md`)

```markdown
# Mozart eval: <YYYY-MM-DD>

**Repos examined**: <names, campaign counts, delta size vs previous run>
**Previous run**: <date or "none — baseline">

## Mechanical metrics (scripts/mozart-lint.sh)

| Repo | Total | status-location | review-drift | duplicate-stages | unclosed-stages | stale-active | stale-paths | stranded-artifacts |
|---|---|---|---|---|---|---|---|---|
| ... | ... | ... | ... | ... | ... | ... | ... | ... |

Trend vs previous run: <deltas, or "baseline — no comparison">

## Fix verification (previous run's targets)

For each verification target the previous report named:
- <target> — MOVED (evidence) | UNMOVED (finding: prose decayed → escalate to mechanical enforcement) | NOT YET MEASURABLE (no post-fix campaigns)

## Findings

Ranked, evidence-cited (file paths, slugs, metrics). Same severity discipline as everywhere else.

## Fixes shipped / proposed

- <fix> — <commit SHA | override path | PR link | field-note proposal> — <metric it should move>

## Cleanup recommendations (consuming repos)

EVAL reports these; it does not execute them. Each is a candidate campaign in that repo.

## Verification targets for the next run

Named, measurable, tied to the fixes above. This section is mandatory.
```

## Cost expectations

The baseline run reads everything and costs the most (the July-2026 baseline across ~285 campaigns took roughly 300k subagent tokens). Subsequent delta runs scale with how many campaigns changed, plus the flat cost of the linter (near-zero) and fix verification. If a run's delta is large, mozart says so and gives a rough estimate before fanning out.
